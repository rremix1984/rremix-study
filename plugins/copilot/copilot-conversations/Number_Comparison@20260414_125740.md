**ai**: 
根据提供的上下文，Karpathy 做 LLM Wiki 的主要目的是：

## 核心理念
大多数人与LLM和文档的体验看起来像RAG（检索增强生成）：每次提问都需要从头重新发现知识，没有积累。Karpathy 的想法是**增量构建并维护一个持久的wiki**—一个结构化的、相互链接的markdown文件集合，它位于你和原始源之间。

## 主要目的

### 1. **解决知识积累问题**
- 传统RAG每次查询都重新推导知识，没有积累
- Wiki是一个持久的、累积的产物，交叉引用已经存在，矛盾已经被标记，综合已经反映了所有阅读内容

### 2. **自动化知识维护**
- 维护知识库的乏味部分不是阅读或思考，而是记账（更新交叉引用、保持摘要最新、标记矛盾等）
- LLM不会感到厌倦，不会忘记更新交叉引用，可以在一次操作中触及多个文件
- 维护成本接近于零，人类只需负责策划源文件、指导分析、提出好问题

### 3. **构建个人知识存储系统**
- 在精神上与Vannevar Bush的Memex（1945）相关
- 创建一个个人的、精心策划的知识存储，具有文档之间的关联路径
- 文档之间的连接与文档本身一样有价值

### 4. **提高知识利用效率**
- 当添加新源文件时，LLM会阅读源文件，提取关键信息，并整合到现有wiki中
- 更新实体页面，修订主题摘要，标记新数据与旧声明相矛盾的地方
- 知识只被编译一次，然后保持最新，而不是每次查询都重新推导

简单来说，Karpathy 的 LLM Wiki 目的是利用 LLM 的自动化能力来构建和维护一个不断累积、自我更新的个人知识库，解决传统知识管理中维护负担过大的问题。

## Wiki 的存储位置

**Wiki** 是一个由 LLM 生成的 **markdown 文件目录**，位于你的个人知识库中。具体来说：

### 三层架构结构：

1. **Raw sources（原始源）**
   - 位置：你精心挑选的源文档集合
   - 内容：文章、论文、图像、数据文件
   - 特点：不可变的，LLM 只读取不修改

2. **Wiki（知识库主体）**
   - 位置：一个专门的目录，存放 LLM 生成的 markdown 文件
   - 内容：摘要、实体页面、概念页面、比较、概述、综合
   - 特点：LLM 完全拥有这一层，创建页面并维护一致性

3. **Schema（模式文档）**
   - 位置：一个配置文件（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）
   - 内容：告诉 LLM wiki 是如何构建的，约定是什么，工作流程是什么
   - 特点：关键的配置文件，使 LLM 成为一个有纪律的 wiki 维护者

### 特殊导航文件：

- **index.md**：wiki 中所有内容的目录，按类别组织
- **log.md**：按时间顺序记录 wiki 的演变过程

## 实际应用建议

虽然上下文没有指定具体的文件路径，但你可以根据需要设置，例如：

```
my-knowledge-base/
├── raw/              # 原始源文件
├── wiki/             # LLM 生成的 wiki 文件
│   ├── entities/     # 实体页面
│   ├── concepts/     # 概念页面
│   ├── summaries/    # 摘要页面
│   ├── index.md      # 索引文件
│   └── log.md        # 日志文件
└── schema.md         # 模式配置文件
```

这种结构让 LLM 能够有效地管理和维护你的个人知识库。
