**ai**: # LLM Wiki 是什么

LLM Wiki 是一种使用大语言模型(LLM)构建个人知识库的创新模式。它不同于传统的RAG(检索增强生成)方法，而是通过增量构建并维护一个持久的、结构化的wiki来积累知识。

## 核心理念

传统RAG方法每次查询都重新发现知识，没有积累。而LLM Wiki的核心区别在于：

- **构建持久Wiki**：创建一个结构化、相互链接的markdown文件集合，位于原始源和用户之间
- **知识积累而非重复推导**：知识只被编译一次，然后保持最新，而不是每次查询都重新推导
- **LLM负责维护**：LLM负责所有繁琐工作(总结、交叉引用、归档、记账)，用户只需负责来源、探索和提问

## 三层架构

1. **原始源层**：不可变的源文档集合(文章、论文、图像等)，作为事实来源
2. **Wiki层**：LLM生成的markdown文件目录(摘要、实体页面、概念页面等)，由LLM完全维护
3. **模式层**：配置文件，定义wiki构建约定和工作流程，使LLM成为有纪律的wiki维护者

## 主要操作

- **摄取**：将新源文件加入集合，LLM阅读、讨论、编写摘要、更新索引和相关页面
- **查询**：LLM搜索相关页面，合成答案，可将答案作为新页面归档到wiki中
- **检查**：定期进行健康检查，查找矛盾、过时声明、孤立页面等

## 实用价值

这种方法论解决了传统知识管理中维护负担过大的问题，使知识能够真正积累和保持最新。适用于个人知识管理、深入研究、阅读书籍、商业团队知识库等多种场景。

简单来说，LLM Wiki就像是你的个人知识库管理员，它帮你阅读、整理、连接所有信息，并保持知识库的更新和一致性，而你只需要专注于提供内容和提出问题。

## 📁 文件夹图标设置功能

### 主要功能

1. **自动提取视频第一帧作为文件夹图标**
   - 脚本会自动从视频文件中提取第一帧画面
   - 将该画面转换为文件夹的图标

2. **生成高清图标**
   - 自动将提取的图像调整为 512x512 分辨率
   - 确保图标在 macOS 中显示清晰

3. **批量操作支持**
   - 支持递归批量设置文件夹图标 (`-r` 参数)
   - 可以一次性处理整个目录树中的所有文件夹

4. **图标移除功能**
   - `-rmicon` 参数可以移除已设置的文件夹图标
   - 恢复文件夹到默认状态

### 技术实现

该功能主要依赖 `fileicon` 工具来实现：
- 使用 `fileicon set` 命令设置文件夹图标
- 使用 `fileicon rm` 命令移除文件夹图标

### 使用场景

这个功能特别适合视频内容管理，例如：
- 为每个用户的视频文件夹 (`DY-用户名`) 设置该用户的第一帧画面作为图标
- 创建视觉化的内容库，便于快速识别不同文件夹的内容
- 提升文件管理的直观性和美观度

### 典型使用命令

```bash
# 为目录中的所有文件夹设置图标
./rename_with_size.sh -icon /path/to/directory

# 递归设置子目录的图标
./rename_with_size.sh -r -icon /path/to/directory

# 移除所有文件夹的图标
./rename_with_size.sh -rmicon /path/to/directory
```

这个功能是脚本中提升用户体验的一个实用特性，通过视觉化的方式让文件夹内容一目了然。

# LLM Wiki 最佳使用实践分析

基于您提供的资料，我将从多个维度分析LLM Wiki的最佳使用实践，帮助您充分发挥这一知识管理系统的潜力。

## 一、核心理念理解与实践

### 1. 从"存储"到"编译"的思维转变

**最佳实践**：
- **区分原始素材与编译产物**：建立`raw/`和`wiki/`两个明确分离的目录
- **理解"编译"与"摘要"的区别**：
  - 摘要：缩短单个文档
  - 编译：将多文档知识重新组织成概念网络
- **接受LLM作为知识维护者**：将繁琐的整理工作交给AI，人类专注于高价值活动

### 2. 知识复利效应的利用

**最佳实践**：
- **每次提问都为知识库增值**：将有价值的分析结果作为新页面归档回wiki
- **建立知识循环**：摄取→查询→归档→再查询，形成知识滚雪球效应
- **鼓励探索性提问**：提出需要跨文档综合的问题，促进知识连接

## 二、架构设计与实施

### 1. 三层架构的最佳实践

**原始源层(Raw Sources)**：
- 保持不可变性，作为事实来源
- 使用一致的命名规范，便于LLM识别
- 定期清理无关内容，保持源质量

**Wiki层(The Wiki)**：
- 让LLM完全负责维护，包括创建、更新、交叉引用
- 使用标准化的页面模板（如frontmatter包含标签、日期、来源）
- 定期进行健康检查，保持结构完整

**模式层(Schema)**：
- 创建详细的配置文件（如`CLAUDE.md`或`AGENTS.md`）
- 明确定义命名约定、工作流程和页面结构
- 与LLM共同演化，根据实际使用调整

### 2. 目录结构设计

**推荐结构**：
```
LLM_Wiki/
├── raw/              # 原始素材
│   ├── reports/      # 报告PDF
│   ├── articles/     # 文章
│   ├── webclips/     # 网页剪藏
│   └── assets/       # 附件（图片等）
├── wiki/             # LLM生成的知识
│   ├── concepts/     # 概念页面
│   ├── entities/     # 实体页面
│   ├── insights/     # 洞察分析
│   └── projects/     # 项目知识
├── index.md          # 全局索引
├── log.md            # 操作日志
└── AGENTS.md         # 配置文件
```

## 三、核心操作的最佳实践

### 1. 摄入(Ingest)操作

**最佳实践**：
- **一次一个源**：保持参与感，阅读摘要，指导LLM强调重点
- **批量处理**：对于大量相似素材，可批量处理以提高效率
- **明确指导**：提供具体的提取要求，如"提取3-5个核心判断，附上论据和我的评注"

**示例提示词**：
```
请阅读raw/reports/中的《2024新能源报告.pdf》，完成以下任务：
1. 创建一篇800字的中文洞察文章，提取3-5个核心判断
2. 如果报告提到电池技术，更新已有的"固态电池"概念页面
3. 更新index.md，添加新页面链接和摘要
4. 在log.md中记录本次摄入
```

### 2. 查询(Query)操作

**最佳实践**：
- **提出需要综合的问题**：如"GPU租赁价格对AI漫剧生产成本有什么影响？"
- **要求多种输出形式**：比较表、分析报告、概念图等
- **鼓励跨文档连接**：引导LLM发现不同素材间的关联

### 3. 检查(Lint)操作

**最佳实践**：
- **定期执行**：每周或每两周进行一次健康检查
- **重点关注**：
  - 页面间的矛盾
  - 过时声明
  - 孤立页面
  - 缺失的交叉引用
- **利用LLM建议**：让LLM推荐需要调查的新问题和源

## 四、辅助文件与工具的最佳实践

### 1. index.md的使用

**最佳实践**：
- **标准化格式**：每页包含链接、一句话摘要、元数据
- **按类别组织**：实体、概念、源等分类
- **LLM自动更新**：每次摄入后自动更新，保持最新

**示例格式**：
```markdown
# 概念
- [[推理成本]] | AI推理的经济成本分析 | 2026-04-01 | 来源: SemiAnalysis, Gartner
- [[GPU租赁]] | GPU租赁市场分析与价格趋势 | 2026-04-02 | 来源: SemiAnalysis

# 实体
- [[OpenAI]] | AI研究公司 | 2026-03-15 | 来源: 多篇报告
```

### 2. log.md的使用

**最佳实践**：
- **标准化格式**：使用`## [日期] 操作 | 内容`格式
- **定期回顾**：用`grep "^## \[" log.md | tail -5`查看最近活动
- **时间线分析**：帮助LLM理解wiki演化历程

### 3. 工具链的最佳实践

**Obsidian Web Clipper**：
- 快速将网页文章转换为markdown
- 设置固定附件文件夹路径
- 绑定下载快捷键，自动下载图片

**图像处理**：
- 设置`raw/assets/`为附件文件夹
- 使用快捷键下载所有引用图片到本地
- 确保LLM能直接访问本地图像

**搜索工具**：
- 小规模时使用index.md足够
- 大规模时考虑qmd等本地搜索引擎
- 利用Obsidian内置搜索功能

## 五、不同场景的最佳实践

### 1. 个人知识管理

**最佳实践**：
- **追踪目标与健康**：将日记、文章、播客笔记归档
- **构建"自我画像"**：随时间积累结构化的个人知识
- **定期反思**：利用wiki进行自我分析和改进

### 2. 深度研究

**最佳实践**：
- **主题聚焦**：选择一个主题进行数周或数月深入研究
- **论文管理**：系统性地阅读和归档论文
- **演化论点**：逐步构建带有演进论点的综合wiki

### 3. 商业/团队应用

**最佳实践**：
- **多源整合**：Slack线程、会议记录、项目文档
- **人类审核循环**：关键信息由人类审核
- **保持更新**：利用LLM做团队不愿做的维护工作

## 六、进阶技巧与注意事项

### 1. 规模管理

**最佳实践**：
- **小规模优势**：几十篇文章、几万字时，index.md足够
- **扩展策略**：超过100个源、数百页时引入更高级检索
- **先跑起来再优化**：不要过度设计，先建立基本流程

### 2. 避免常见陷阱

**最佳实践**：
- **不要过度依赖**：wiki是起点不是牢笼，结合外部搜索
- **保持人类判断**：LLM可能产生幻觉，需要人类审核
- **定期备份**：使用Git等工具进行版本控制

### 3. 效率优化

**最佳实践**：
- **批量操作**：相似素材可批量处理
- **模板复用**：建立标准页面模板，提高一致性
- **自动化**：利用CLI工具简化重复操作
